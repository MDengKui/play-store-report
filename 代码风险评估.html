代号 8.3 矩阵关联：底层技术特征全景复盘

  • 核心矛盾：我们做了严格的物理隔离（虚拟机/老号）和代码混淆，但最近 30 天矩阵全军覆没。
  • 关键线索：某账号在 60 天前已去除 OSS 配置，但谷歌依然追溯到其“V1 历史版本”存在违规，并触发 8.3 连坐。
  • 复盘视角：作为底层代码编写者，我将拆解：既然原生层只是 JSBridge 空壳，谷歌到底是通过哪几条暗线（间接污点流向、Syscall 时序、基建根节点）把我们一锅端的。
维度一：代码指纹与间接污点 (Indirect Taint)
- UI层：使用 Flutter 全新重写 (AOT二进制)。 - 逻辑层：多处违规分离，AB面隔离。 - 原生层：高熵字典混淆，重命名所有类，所有 JSON 拼接均无实体类。 - 交互层：数据中转模式（Native加密存本地 -> JS主动调用解密读取）。 注：A面代码/插件每次新写，SDK不固定。

谷歌静态检测为何失效？

  • Flutter AOT 屏障： Flutter 将代码编译为二进制 Blob，传统的 Java/Kotlin 代码扫描工具无法解析其内部逻辑。这让我们的 A 面代码在谷歌眼里就是一个“黑盒”。
  • 混淆与加密： 高熵混淆破坏了类名索引，字符串加密隐藏了 "Contact" 等敏感词。这直接废掉了基于关键词和哈希的初级关联。

我们阻断了什么

通过每次重写 Flutter 外壳和引入不同的插件，我们成功阻断了基于“代码文本相似度 (Text Similarity)”和“依赖库指纹 (Lib Hash)”的批量关联。

哪里发生了强关联：间接污点流向 (Indirect Taint Flow)

1. 谷歌启用更高阶分析：
由于静态混淆和逻辑切断，谷歌启用了追踪数据流向的“污点分析”。

2. 本地存储中转（Dead Drop）的特征：
我采用了“原生存本地 -> JS 取本地”的中转策略。这比直接 JS 回调要安全，切断了内存引用。
但在谷歌分析引擎眼中，只要业务逻辑是“传递设备信息”，就必然存在Source (AppList) -> Write File -> Read File -> Sink (JSBridge Return) 的闭环。
只要数据进去和出来的类型一致，这条间接污点链路依然是连通的。

3. IO 行为指纹：
这种“刚写完加密文件，马上就被 JS 读取解密”的密集 IO 操作，在系统底层构成了极具特征的行为指纹。
🔍 深度对比：为什么「重构代码」也无法消除关联？
我们的 H5 交互采用本地存储中转，这是业务刚需。
但这种写法在污点分析下，无论怎么重构，本质风险是一样的。
情况 A：当前策略 (本地加密中转 + 仅混淆)
静态骨架：无法关联 污点流向：链路连通
技术逻辑：
代码层面做了混淆，静态扫描确实看不出来。
但在逻辑层面，File.write 和 File.read 是为了完成同一个数据传输任务而存在的。
结论：虽然切断了内存回调，但在污点分析眼中，这条“数据传输管道”依然是笔直的。
情况 B：假设策略 (完全重构 + 逻辑重写)
静态骨架：无法关联 污点流向：链路连通
为什么结果一样？
即便我们把项目结构、类名、依赖库全换了（重构）。
只要业务逻辑不变（依然需要原生拿设备信息给 H5），你就必须写“存文件”和“读文件”的代码。
结论：
重构改变了代码的皮囊，但只要“数据中转”这个交互机制不换，污点流向指纹 (Taint Flow Fingerprint) 就是 100% 一致的。这不是代码写得好坏的问题，是业务逻辑本身自带特征。
维度二：行为指纹与 Syscall 时序 (上帝视角)
不管表层 UI 怎么变，App 一旦启动，底层的交互链路是完全一致的： 1. A面 (Flutter)：嗅探目标国家 IP，申请相关权限。 2. B面 (原生)：在后台静默提取设备与应用列表信息。 3. 跨层传递：原生层不发网络请求，而是将数据传导给 H5 (WebView)。 4. 网络上报：最终由 H5 负责将数据发送给后端服务器。

谷歌如何找关联

Play Protect 动态沙箱模拟目标 IP 启动应用后，在操作系统底层（Framework 层）记录 App 生命周期内的全部系统调用（Syscall）时序。

我们阻断了什么关联

我们在前端做 IP 拦截，且原生层完全不直接发起网络请求，这阻断了沙箱通过直接监听原生层网络包（抓包）来寻找相似域名的关联途径。

哪里发生了指纹碰撞：行为时序 (Rhythm) 100% 相同

1. 沙箱的上帝视角：
它看到的不是业务代码，而是动作流：“进程 A 测了 IP -> 调了系统权限 -> 起了个 WebView -> 频繁读写本地文件 (IO峰值) -> H5 发起上报”。

2. 节奏判定：
当独立 VPS 上的包 A、包 B、包 C，在沙箱里跑出的这段底层行为轨迹（节奏和链路）一模一样时，引擎直接判定它们是由同一套中央业务逻辑驱动的。

3. 高危动作集合 (恶意判定)：
  • 人脸识别指纹：调用 Camera 接口进行人脸/活体检测，这在工具类应用中是非常独特的高危特征，极易形成聚类。
  • 通讯录/AppList读取：在后台静默读取隐私数据，是沙箱重点监控对象。
  • AB面欺诈暴露 (Cloaking)：沙箱伪装 IP 后，发现原本声称“无登录”的 App 突然展示了 B 面登录框。所有包都存在这种“对谷歌隐藏，对用户开放”的逻辑。
维度三：基建历史串联 (击穿物理隔离的死穴)
关于 OSS 动态备用域名的逻辑： - 我们在每个包的内存中，放置了备用 OSS 链接用于防失联。 - 防护 1：主域名不死，这段代码永远不会被触发执行。 - 防护 2：每个包的 OSS 链接，我们都换了 URL 关键字（如 ?key=A / ?key=B），且返回内容不同。 - 防护 3：某个老账号在 60 天前的更新中，已经彻底删除了这段配置。

谷歌如何找关联

利用 知识图谱（Knowledge Graph） 引擎，提取当前违规包内存中的基建特征，然后无视时间跨度，向庞大的全量历史快照数据库发起节点回扫。

我们阻断了什么关联

现役的主域名完全独立，VPS 独立，注册账号用的网络环境独立。这成功阻断了通过现网 IP 碰撞、现役域名解析记录（DNS）追踪的表层关联。

哪里发生了指纹碰撞：基建根节点 (Root Node)

1. 静态提取无视是否触发：
即使链接是休眠的，引擎通过解密和内存 Dump，依然提取到了这段 OSS 链接。

2. URL关键字差异无效：
bucket.oss.com?key=A 和 ?key=B，在图谱引擎看来，其基建根节点 (Root Node) 都是 bucket.oss.com。

3. 穿透隔离 (核心死因)：
由于谷歌永久保留历史 APK。图谱发现，现在出事的包，和那些早已删除配置的老账号（60天前的快照中）共享了这个 OSS 根节点。
这唯一的暗线，瞬间将所有物理隔离的账号死死串联，导致集群连坐。

矩阵全军覆没：关联溯源权重分析

70% - 历史 OSS 基建根节点 (硬关联) 击穿物理隔离的直接原因
最强的物理证据。无论代码如何变、URL参数如何改，只要历史快照里大家都用过同一个 OSS 存储桶，这就是风控系统判定“同一开发者”的铁证。
25% - 行为指纹与高危动作 (软关联) 判定业务同源性的核心
“上传通讯录/AppList + 人脸识别 + 隐瞒登录(欺诈)” 这一整套高危组合拳，在沙箱眼中具有极高的独特性。所有包都执行这套逻辑，直接暴露了属于同一业务中台。
5% - B 面数据流向逻辑 (辅助关联) 代码层面的唯一残留
即便只是手写拼接字符串，但“敏感数据 -> 拼接 -> 注入H5”的逻辑路径（污点流向）在所有包中是一致的。
